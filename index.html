Engineering Notes

Honest technical reflections from projects and learning.

How I Think About System Design as a Student

System design isn't just for backend interviews. Every project has architecture. The difference between code that scales and code that breaks is how well you think about layers, separation of concerns, and trade-offs. I approach every project asking: "What would happen if this had 10x more users?"

What Building a City Simulator Taught Me About Backend Systems

A simulator is really just a backend system with a frontend visualization. The core lesson: data flows in, rules process it, state updates, clients consume the result. Understanding this makes me better at building real services.

Why I Care About Clean Architecture More Than Frameworks

Frameworks are tools. They change every 3 years. Architecture is timeless. If I can write clean code in vanilla JavaScript, I can write clean code anywhere. I focus on principles first, frameworks second.

Common Backend Mistakes I've Made and Fixed

Mixing concerns: I used to write business logic inside request handlers. Now it's always separated. Tests are easier. Bugs are easier to isolate.

Premature optimization: I optimized code that didn't matter. Profile first, optimize after. Measure before and after.

No versioning plan: APIs change. Build versioning from day one.

On Debugging Intuition

Good debugging is about reducing the search space. Break the problem in half. Check invariants. Use immutability to make state transitions explicit. The faster you can reproduce a bug, the faster you fix it.

Learning Through Building vs. Learning Through Tutorials

Tutorials teach you syntax. Building teaches you systems. I choose projects that force me to think about architecture, not just code. That's where growth happens.
