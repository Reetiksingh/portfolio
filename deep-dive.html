3D City Simulator — Deep Dive

Why I Built This

I wanted to understand how complex systems actually work—not from tutorials, but by building something real. A city simulator forced me to think about:

  • How state changes propagate through a system
  • How rules interact with data
  • How to separate concerns in a larger codebase
  • How frontend and backend can work together

Simple tutorials teach you syntax. Building a simulator teaches you systems thinking.

System Architecture

The simulator is built as distinct layers:

Rendering Layer

Three.js, camera, visualization

Frontend Logic Layer

Input handling, event delegation

State Management

Centralized city state, immutable updates

Simulation Engine

Rules, scoring, city growth logic

This separation means changing the rendering layer doesn't break the engine. The engine could theoretically be ported to a backend service without core logic changes.

Backend & System Logic (Core)

City State: Buildings, population, resources, and infrastructure stored as structured data. Updates are immutable—old state is never mutated, new state is computed.

Rules Engine: Zoning affects growth. Infrastructure affects capacity. Rules are defined declaratively, not scattered through rendering code.

Scoring: Dynamic score based on population, efficiency, and growth. Changed as the simulation runs.

Scalability: If this were a real service, the engine could be extracted to a backend microservice, with the frontend consuming simulation state via API.

Design Decisions

Why immutable state? Easier to debug, easier to reason about. If something breaks, I can rewind time.

Why separate the engine from rendering? I can test the engine without Three.js. I can swap renderers without breaking logic.

Why not use an existing game engine? I wanted to understand every decision. Frameworks hide complexity; that's useful in production, but not for learning.

Trade-offs & Future Improvements

Current bottleneck: All state updates run on the main thread. At scale, this would block rendering.

Solution: Move the simulation engine to a Web Worker. Frontend becomes a pure rendering layer.

At production scale: The engine lives on a backend server. Clients send actions, receive state deltas via WebSocket. Multiplayer becomes possible.

Engineering Learnings

Bug that changed my thinking: I mixed state updates with rendering logic. When a bug appeared, I couldn't tell if it was logic or rendering. Separating them immediately made debugging 10x faster.

Performance mistake: I was creating new objects for every frame. This caused garbage collection spikes. Switching to object pooling fixed it.

Architectural lesson: Clean separation of concerns isn't a nice-to-have—it's what makes systems survivable. Code that works at 100 lines is chaos at 10,000.
